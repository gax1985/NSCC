Denial of Service (DoS)

In Week 7 we will go over a very well known attack- denial of service (DoS). The most popular form of this attack is probably Distributed Denial of Service (DDoS), which involve the coordination of a huge number of devices to make request to a web service and eventually "overflooding" the targeted server.

In fact, DoS can appear in many forms, In chapter 14 of the Web Security book, several types were discussed, including DDoS, regex DoS (ReDoS) and Logical DoS. All these attacks have a shared goal- disturb the normal service provided.

Regular Expression DoS (Regex DoS)

To understand  what Regex DoS is, it is important to understand what Regular Express is. In a nutshell, regular expression is a way for identifying patterns from characters and strings. 

This page gives a lot of good example of how regular expression can be used for searching and matching different patterns.
https://support.google.com/a/answer/1371417?hl=en

For example, if we want to look for any IP address within the range 192.168.1.0 to 192.168.1.255 in a log file, we can just use the "Find/Search" function in a text editor. However, if we want to do it programmatically, regular expression will be a very powerful tool as shown below (from the support.google.com page)

Match Any IP Address in a Range
Usage example Match any IP address within the range 192.168.1.0 to 192.168.1.255.
Regex examples Example 1:   192\.168\.1\.
Example 2:   192\.168\.1\.\d{1,3}
Notes
  • The \ before each period “escapes” the period—that is, it indicates that the period isn't a regex special character itself.
  • In Example 1, no characters follow the last period, so the regex matches any IP address beginning with 192.168.1., regardless of the number that follows.
  • In Example 2, \d matches any digit from 0 to 9 after the last period, and {1,3} indicates that the digits 1 to 3 can appear after that last period. In this case, the regex matches any complete IP address beginning with 192.168.1.. This regex also matches invalid IP addresses, such as 192.168.1.999.

https://regexr.com/

How does this useful tool becomes a potential vulnerability?

While regex is a powerful and usually efficient tool in searching for patterns. Some search pattern can cause a huge toll on the processing time when handling some patterns.

In the book it talks about how a simple validation on the pattern ab can cause an issue when the entered string was ababababa , the last character a make the string invalid when being checked with regex, but every single possible pattern has to be checked recursively which cause an exponential (50 -100x) time for processing. 

Regex DoS attack is hard to identify as a regex vulnerability is hard to detect and may not happen until someone carefully craft an inquiry on a specific service.

Logical DoS 

Logical DoS can appear in almost anywhere in the system. It covers all potential exploit of legitimate service usage which cause the drainage of available resources, this will make the service unavailable to other users, i.e. Denial of Service.

The book listed out the following operations
• Any operation you can confirm operates synchronously
• Database writes
• Drive writes
• SQL joins
• File backups
• Looping logical operations

Distributed DoS (DDoS)

Lastly, one of the most famous DoS attack is DDoS. A classic example was a group of user simply keep hitting the refresh key and make numerous request to refresh a page in a given time. This can certainly be executed programmatically by a simple javascript (https://www.w3schools.com/JSREF/met_loc_reload.asp). If the web service did not implement any filtering or protection for DDoS attack, the web server can easily be taken offline or become extremely slow due to this kind of attack.

Additionally, by using other security vulnerabilities (i.e. broken security access, viruses...), a mass number of computers/devices can be compromise and become part of a bot net (zombie network), which may be used for attack either internally or externally.
