About Cross Site Scripting (XSS)

Cross Site Scripting, as the name suggests is also a type of injection attack, specifically describing the vulnerability caused by the execution of script to a website, and these scripts are being executed on another user's computer/account.

A typical SQL attack is caused by users' input not properly filtered/scrutinized/sanitized and is being executed on the server. Imagine if this injection does not affect the servers, but other users, this is what a cross-site scripting different from other forms of injection attack. XSS usually target cookies, session tokens or other sensitive information stored on the victim's browser.

In short, XSS attacks are usually launched via

  • (Input/injected) Data received from the client side from an untrusted source, usually via web API (e.g. The GET/POST request discussed few weeks ago)
  • (Output) The data injected is sent to other users without being validated for malicious content

There are three main types of XSS attacks (Web Application Security book and OWASP website)

  1. Stored (the code is stored on a database prior to execution)
    A classic example of a stored XSS attack is in a discussion forum, a user injected some javascript code and is stored on the server. When other users view the content of the discussion thread, this script is being sent to the visitors and is executed. The code can be as simple as an alert message box or other malicious code.
    As the name suggested, the code is being stored on the remote server and is being sent to other users.
  2. Reflected (the code is not stored in a database, but reflected by a server)
    Reflected XSS attack usually starts with some phishing technique. Victims receive an email/link which looks like it is from a trusted source. When the user access (e.g. click on the link/submit a form), the receiving end will then be considered as a "trusted" server and thus an attack can be launched.

    From https://brightsec.com/blog/reflected-xss/

    Consider a social media site that requires users to authenticate to send and view messages. The website has a search function which displays the search string in the URL, like this:

    http://socialize.com?query=latest&news

    An attacker notices this and tries the following string as their search:

    <script type='text/javascript'>alert('test');</script>

    If the website does not properly sanitize inputs, this test script will appear in the URL, like this:

    http://socialize.com?query=<script type='text/javascript'>alert('test');</script>
